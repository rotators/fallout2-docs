LST File Format

An LST file is a plain text table where each physical line corresponds to one numbered entry. The first line is entry 0, the second line is entry 1, and so on. These indexes are used by several Fallout data structures to refer to files indirectly.

The data on a line is usually a single token, most often a file name or base file name. Fallout data files typically use DOS-style CRLF line endings, but readers should not rely on the line ending alone to identify the format.

Comments are allowed and start with a semicolon (;). For art list files, the engine reads only the first token on each line and stops at a space, comma, semicolon, tab, carriage return, or newline. This means an inline comment can safely follow the entry value.

Blank lines and lines containing only comments should not be used. They still occupy a line position, so they can shift every following index and break references from other files.

General rules

Rule Description
Indexing Indexes are zero-based. Entry 0 is the first physical line in the file.
Ordering Order is part of the data. Reordering lines changes every reference that stores a line index.
Comments Semicolon comments are common. Some specialized lists also use other trailing metadata, so preserve text after the first token when editing.
Case Original data usually uses uppercase filenames, but tools should avoid changing case because unpacked files on modern systems can be case-sensitive.
Extensions Some lists store complete filenames such as STRENGTH.FRM; others store base names that the engine combines with an extension.

Example

Here are the first five lines from skilldex.lst. They correspond to entries 0 through 4:

STRENGTH.FRM  ; Strength     (Basic Stat)
PERCEPTN.FRM  ; Perception   (Basic Stat)
ENDUR.FRM     ; Endurance    (Basic Stat)
CHARISMA.FRM  ; Charisma     (Basic Stat)
INTEL.FRM     ; Intelligence (Basic Stat)

Art lists

The art system loads one LST file for each art object type from paths of the form art\TYPE\TYPE.lst. Examples include art\items\items.lst, art\critters\critters.lst, art\tiles\tiles.lst, and art\skilldex\skilldex.lst.

For these art lists, the line number becomes the art index stored in a FID. The art type selects which LST is used, and the low bits of the FID select the line within that list. For example, a skilldex art FID can point at entry 0 of skilldex.lst, which resolves to STRENGTH.FRM.

Fallout 2 CE stores art list names in fixed 13-byte slots: up to 12 characters plus a trailing null byte. Tools that generate art LST files should keep base names within that limit unless they know the target engine supports longer names.

Art FID layout

Art FIDs pack several fields into one 32-bit value. The low 12 bits store the LST index. Bits 24 through 27 store the art object type, which selects the LST file. Critter FIDs also use bits 12 through 15 for weapon animation, bits 16 through 23 for animation type, and bits 28 through 30 for rotation.

bits 28..30  rotation
bits 24..27  art object type
bits 16..23  animation type
bits 12..15  weapon animation code
bits  0..11  LST index

The engine constructs art FIDs with the following layout:

fid = ((rotation << 28) & 0x70000000)
    | (objectType << 24)
    | ((animType << 16) & 0x00FF0000)
    | ((weaponCode << 12) & 0x0000F000)
    | (lstIndex & 0x00000FFF);
Field Mask Description
LST index 0x00000FFF Zero-based line number within the selected art LST.
Weapon animation 0x0000F000 Critter weapon animation code. Usually zero for non-critter art.
Animation type 0x00FF0000 Critter animation or head fidget/expression selector, depending on art type.
Art object type 0x0F000000 Selects the art LST family, such as items, critters, tiles, or skilldex.
Rotation 0x70000000 Direction used for directional FRM/FR0..FR5 art. Not every art type uses it.

For example, a non-animated skilldex image at entry 0 uses object type 10 and LST index 0, giving an art FID of 0x0A000000. The object type selects art\skilldex\skilldex.lst, and index 0 selects STRENGTH.FRM.

Non-critter art usually leaves the animation, weapon, and rotation fields at zero. The important part for LST editing is that changing a line number changes the value expected in the low 12 bits of any FID that references it.

Index limits

Because the art index field in a FID is 12 bits wide, a normal art FID can directly address entries 0 through 4095 within a single art LST. Adding more lines than that may still produce a text file, but those entries cannot be represented by the standard FID layout.

Critter and head lists are special. The first token is still the art base name, but the remaining comma-separated fields are read as metadata. In critters.lst, extra fields can define an alias critter and whether that critter should run. In heads.lst, extra fields describe the number of good, neutral, and bad fidgets.

Filename construction

Most art lists store filenames that are used directly under their art type directory. Critter and head lists are different: their entries are base names. For critters, the engine appends animation and weapon animation codes and then chooses .frm or a directional .frN extension. For talking heads, the engine appends fidget and expression codes before adding .frm.

This is why a critter list entry such as hmjmps does not name a complete file by itself. It is the stem used to build filenames for that critter's animation set.

Localized art lookup

When the game language is not English, the art loader can try a localized art path before falling back to the normal path. The LST index still selects the same entry; only the resolved file path changes.

Special cases

The script system uses indexes from scripts.lst. Script ids and proto fields can refer to entries in that list, so inserting or deleting lines can change which script an object uses.

Entries in scripts.lst are expected to contain an .int filename. Fallout 2 CE stores the script base name without the extension and rejects names longer than 13 characters before .int. A line can also contain local_vars=N metadata after a # marker; the script engine uses that value to allocate local variables for the script.

arroyo.int        # local_vars=4
doorlook.int      # local_vars=0

Stored script references use a packed value of the form 0x0Y00XXXX, where Y is the script type and XXXX is the zero-based index in scripts.lst. Known script type values are 0 for system, 1 for spatial, 2 for time, 3 for item, and 4 for critter. A value of -1 means no script.

Editing notes

When adding new entries, append them to the end of the list whenever possible. This preserves existing indexes. If a line must be removed, replacing it with a harmless placeholder is usually safer than deleting it, because deletion shifts every later index.

When building tools, treat LST files as indexed tables rather than unordered name lists. A sorted or deduplicated LST can be technically valid text but still invalid game data.

Preserve formatting unless there is a reason to rewrite it. Line endings, comments, case, trailing metadata, and placeholder lines may all carry practical value for mod compatibility even when the engine only reads the first token.

Parser pitfalls

The art list loader counts every physical line before it extracts tokens. A blank line is therefore still an entry; after tokenization it becomes an empty filename. A line that begins with a delimiter such as a space, comma, semicolon, or tab also produces an empty first token.

For art lists, the loader copies at most 12 characters from the first token and then appends a null byte. Longer names are silently truncated in the in-memory art table, which can make two different entries collide if their first 12 characters are identical.

Because comments are parsed differently by different list consumers, semicolon comments are safest for simple art lists, while scripts.lst uses # local_vars=N metadata. A generic LST editor should not assume one comment convention is valid for every list.
